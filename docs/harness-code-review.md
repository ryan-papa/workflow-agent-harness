# Harness: 코드 리뷰 기준

## 평가 항목 (7개, 각 10점)

### 1. 정확성 (Correctness)

| 점수 | 기준 |
|:----:|------|
| 9~10 | 엣지 케이스(null, 경계값, overflow) + 동시성 + 장애 시나리오까지 처리 |
| 7~8 | 핵심 로직 정확, 실패 경로(타임아웃, 네트워크 단절 등) 일부 누락 |
| 5~6 | 메인 시나리오만 동작, 예외 시 NPE·상태 전이 오류 가능 |
| 3~4 | 특정 조건에서 명백한 버그, 데이터 정합성 위험 |
| 1~2 | 핵심 기능 미동작 또는 regression |

> **8점 질문:** null이면? 동시 요청이면? 외부 API 5초 무응답이면? → 답이 코드에 있으면 8+

### 2. 설계 및 구조 (Design & Architecture)

| 점수 | 기준 |
|:----:|------|
| 9~10 | 단일 책임, 올바른 의존성 방향, 수정 없이 확장 가능, 도메인·인프라 분리 |
| 7~8 | 구조 양호, 일부 책임 혼재 (서비스에 표현 로직 등) |
| 5~6 | 레이어 경계 모호, 하나의 클래스가 3개 이상 역할 |
| 3~4 | God 클래스, 순환 의존성, 로직 중복 산재 |
| 1~2 | 구조 부재, 수백 줄 절차적 나열 |

> **8점 질문:** 가상의 새 요구사항 추가 시 기존 코드 수정 없이 새 클래스만 추가하면 되는가?

### 3. 가독성 (Readability)

| 점수 | 기준 |
|:----:|------|
| 9~10 | 이름만으로 흐름 파악, 주석은 "왜"만, 함수 ≤20줄, 들여쓰기 ≤3단계 |
| 7~8 | 대체로 명확, 모호한 이름 1~2개 또는 복잡한 조건문 소수 |
| 5~6 | 약어 남발, 함수 40줄+, 중첩 if 4단계+, 이해 멈추는 구간 2회+ |
| 3~4 | 변수명 a/b/x, 주석 `// 처리함` 수준, 작성자에게 직접 물어야 이해 |
| 1~2 | 작성자도 한 달 뒤 이해 불가 |

> **8점 질문:** 아무 함수의 이름만 읽고 입력·동작·결과를 유추할 수 있는가?

### 4. 테스트 품질 (Test Quality)

| 점수 | 기준 |
|:----:|------|
| 9~10 | Happy + Sad Path 커버, 구현 아닌 행위 검증, 테스트명이 스펙 문서 역할 |
| 7~8 | 주요 경로 충실, 경계값·실패 시나리오 일부 누락 |
| 5~6 | Happy Path만, Mock 과도 사용, 커버리지 수치만 높은 무의미한 테스트 |
| 3~4 | 테스트 깨짐 또는 @Disabled, 외부 환경 의존으로 수시 실패 |
| 1~2 | 테스트 없음 또는 `assertTrue(true)` |

> **8점 질문:** 로직 함수의 `>`를 `>=`로 바꿨을 때 테스트가 실패하는가?

### 5. 보안 (Security)

| 점수 | 기준 |
|:----:|------|
| 9~10 | 입력 전부 검증, Injection·XSS·CSRF 방어, 인증/인가 정확, 민감정보 설정 분리, 로깅에 개인정보 미노출 |
| 7~8 | 주요 보안 처리됨, 입력 길이 미제한·스택트레이스 노출 등 방어적 검증 일부 누락 |
| 5~6 | 인증은 있으나 인가 누락 엔드포인트 존재, 입력이 검증 없이 쿼리에 결합 |
| 3~4 | 하드코딩 API 키, 패스워드 평문 저장, 인증 우회 가능 |
| 1~2 | 인증 없이 관리자 기능 접근 가능 |

> **8점 질문:** 모든 외부 입력에 `'; DROP TABLE users;--` 넣어도 안전한가?

### 6. 성능 및 효율성 (Performance)

| 점수 | 기준 |
|:----:|------|
| 9~10 | 복잡도 최적, N+1 없음, 적절한 캐싱, 대용량 시 스트리밍/배치/페이지네이션 |
| 7~8 | 현재 규모 문제없음, 10배 증가 시 병목 1~2개 잠재 |
| 5~6 | 루프 안 DB 호출(N+1), 전체 테이블 스캔, 무한 증가 컬렉션 메모리 유지 |
| 3~4 | O(n²)이 핫 패스에 존재, 커넥션/리소스 누수 가능 |
| 1~2 | 프로덕션 배포 시 즉시 장애 |

> **8점 질문:** 데이터 100배 증가 시 버티는가? 또는 페이지네이션/캐싱으로 대응 가능한가?

### 7. 유지보수성 및 컨벤션 (Maintainability)

| 점수 | 기준 |
|:----:|------|
| 9~10 | 컨벤션 100% 준수, 커밋 의미 단위 분리, 설정 외부화, 로깅 적절, 기존 패턴과 일관 |
| 7~8 | 컨벤션 대부분 준수, 커밋 약간 뭉뚱그려짐, 로깅 일부 과부족 |
| 5~6 | 스타일 혼재, PR에 관련 없는 변경 혼입, 커밋 메시지 "fix" 수준 |
| 3~4 | 기존과 다른 패턴 독자 도입, 불필요한 라이브러리, 설정값 하드코딩 |
| 1~2 | 린트 에러 무시, 빌드 경고 방치, 주석 처리 코드 대량 방치 |

> **8점 질문:** 처음 보는 동료가 기존 코드와 구분 못 할 정도로 일관되는가?

## 판정

**통과 조건 (모두 충족):**

| 조건 | 기준 |
|------|------|
| 평균 | ≥ 8.0 |
| 각 항목 최저 | ≥ 7 |

최저 미달 시 평균 8.0 이상이어도 **미통과**.

**재시도:** 미통과 → Dev Agent 재투입 (최대 3회). 3회 후 < 7.0 → 사용자 판단, 7.0~8.0 → 통과 처리.

**이슈 처리:** High/Mid → Dev Agent 자동 수정. Low → 테이블 제시, 사용자 선택.

## PR 유형별 포커스

| PR 유형 | 집중 영역 |
|---------|-----------|
| 버그 픽스 | 정확성 + 테스트 |
| 새 기능 | 설계 + 보안 + 성능 |
| 리팩토링 | 가독성 + 설계 + 테스트 |
| 인프라·배포 | 보안 + 유지보수성 |
