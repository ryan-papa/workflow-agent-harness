# rp-specify

도메인 → 기술 구체화. 질문으로 요구사항을 명확히 한다.

## 트리거

- 프로젝트 초기화 완료 후
- `/rp-specify` 명령

## 질문 원칙

**Do:**
- 구체적 장면을 상상하게 만드는 질문
- 해결책이 아닌 문제·맥락을 묻는 질문
- 가설을 꺼내는 여백이 있는 질문("만약에~"로 숨은 제약 발견)
- 답을 다시 돌려주기("A라는 뜻 맞나요?"에서 진짜 정보가 나옴)

**Don't:**
- 해결책을 묻는 질문("댓글 기능 넣을까요?") → 의뢰인에게 설계를 떠넘김
- 추상적인 질문("비전이 뭐예요?") → 교과서 같은 답만 나옴

## 절차

1. 사용자 요청 분석 → 도메인 영역 파악
2. **5단계 질문** 순서대로 1개씩 대화형 진행
3. 후속: 화면·기능 구체화 → 시장 조사 → 기술 정보 → 구현 조건 검증
   - **대안 탐색**: A/B/C/D(현상 유지 포함) 대안 + 장단점·공수·판단 근거 유형(창업자 직감·엔지니어 선호·제품 가설) 수집
   - **실패 모드**: 코드경로 × 시나리오 × 감지 방법 × UX 대응 표 수집
   - **톤·정체성**: 톤·어투·금칙어·레퍼런스·샘플 카피 수집
   - **AI 기능 포함 여부 확인**: 포함 시 정답 샘플셋·채점 기준·프롬프트 주입 방어·합격선 수집
   - **사용자 검증 게이트**: 실사용자 반응 검증 방법(프로토타입·인터뷰·A/B 테스트·내부 시범 사용) 수집
4. 재귀적 구체화: 새 기능 발견 시 동일 절차 반복
5. 필요 시 리서치 수행 → `docs/research/` 저장

## 5단계 질문 구조

| 단계 | 묻는 것 | 대표 질문 |
|:----:|---------|----------|
| 1. 존재 이유 | 왜, 지금 왜 | "지금 이게 없어서 어떤 일이 안 풀리고 있나?" |
| 2. 사용자 | 누가 | "가장 자주 쓸 사람 한 명을 상황으로 묘사해주세요" |
| 3. 핵심 장면 | 무엇을, 어떻게 | "가장 자주 일어날 장면 3개를 순서대로요" |
| 4. 경계 | 무엇이 아닌가 | "'이런 느낌은 아니다' 싶은 레퍼런스가 있나?" |
| 5. 성공 기준 | 어떻게 알까 | "3개월 뒤 어떤 숫자/장면을 보면 성공? 접을 기준은?" |

## 구체화 품질 기준

각 항목이 가져야 할 4가지 성질:

| 성질 | 설명 |
|------|------|
| 관찰 가능 | 근거가 있다 |
| 반증 가능 | 확인할 방법이 있다 |
| 경계가 있다 | 포함/비포함이 분명하다 |
| 분기가 있다 | 예외·에러·빈 상태가 고려되어 있다 |

**리트머스:** 서로 다른 두 사람이 이 문서만 보고 작업했을 때 같은 방향을 향하는가?

## 질문 형식

CLAUDE.md의 Interaction 규칙에 따른다:
- 1개씩 대화형 진행
- 객관식 4개 + 추천 점수(10점 만점) + ★ 추천 표시
- 주관식 항목 별도 제공

## 완료 조건

- 5단계 질문 + 후속 항목(화면·기능 구체화·시장 조사·기술 정보) 완료
- 모든 기능이 4가지 성질 충족
- 리서치 필요 시 결과 저장 완료
