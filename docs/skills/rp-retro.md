# rp-retro

배포 완료 후 하네스 워크플로우 회고.

## 트리거

- `/rp-ship` 완료 직후 자동 진행
- `/rp-retro` 명령으로 수동 실행 가능

## 점검 범위

| 영역 | 내용 |
|------|------|
| 절차 준수 | 각 단계 스킵 여부, 순서 위반, 규칙 미준수 |
| 효율성 | 병목 구간, 불필요한 반복, 더 나은 방법 |
| 규칙 개선 | 하네스 규칙 자체의 개선 필요성 |

## 절차

### Phase 1: 영역별 에이전트 평가

각 영역 전문 에이전트가 담당 파트를 독립 평가:

| 에이전트 | 담당 영역 | 평가 대상 |
|---------|----------|----------|
| 개발 에이전트 | 개발 파트 | 태스크 분해, 브랜치 전략, 코드 품질, 테스트 |
| QA 에이전트 | 테스트 파트 | QA 사전 검증, 기능 QA, 콘텐츠 검수, E2E |
| 디자인 에이전트 | 디자인 파트 | 디자인 QA, 접근성, 반응형 (UI 프로젝트만) |
| 리뷰 에이전트 | 리뷰 파트 | 코드리뷰 실효성, 이슈 처리 적절성 |
| 배포 에이전트 | 배포 파트 | CI, 머지 절차, 산출물 보고, 배포 안전성 |

각 에이전트가 담당 영역 평가 항목을 10점 만점으로 채점 + 이슈·개선점 기술.

### Phase 2: 최종 취합 + 보고

1. 영역별 평가 결과를 종합해 전체 테이블 작성
2. 위반·비효율 원인 교차 분석 (영역 간 영향 파악)
3. 하네스 규칙 자체 개선 제안 도출
4. 결과를 사용자에게 보고
5. 개선 제안을 번호형 선택지로 제시
6. 사용자 승인된 항목만 하네스 문서에 반영

## 평가 항목 (각 10점 만점)

| 항목 | 설명 |
|------|------|
| 절차 준수 | 하네스 단계를 순서대로 빠짐없이 수행했는가 |
| 문서 선행 | 코드 전에 PRD/문서가 먼저 업데이트되었는가 |
| 테스트 충실도 | 테스트 작성·통과 규칙을 지켰는가 |
| QA 완전성 | QA/콘텐츠 검수가 누락 없이 수행되었는가 |
| 코드리뷰 품질 | 코드리뷰가 형식적이지 않고 실질적이었는가 |
| 배포 안전성 | CI 통과 후 머지, 사용자 승인 절차를 지켰는가 |
| 효율성 | 불필요한 반복·대기·수동 작업 없이 진행했는가 |
| 커뮤니케이션 | 산출물 보고·중간 보고가 적절했는가 |

## 결과물 형식

### 1. 단계별 평가 테이블

```
| 단계 | 점수 | 준수 | 비고 |
|------|:----:|:----:|------|
| [1] 초기화 | 10/10 | ✅ | — |
| [8] QA | 5/10 | ❌ | QA 생략, 빌드 확인만 수행 |
| ... | | | |
| **평균** | **X.X/10** | | |
```

### 2. 개선 제안 (번호형 선택지)

```
이번 회고에서 도출된 개선 제안:

1. [규칙 추가] 콘텐츠 프로젝트도 QA 자동 진행 필수화 — 9/10 ★ 추천
2. [효율 개선] 빌드 검증과 렌더링 확인을 병렬로 진행 — 7/10
3. [규칙 변경] ... — 6/10
4. 전체 반영
5. 반영 안 함
```

사용자가 번호 선택 → 해당 항목만 하네스 문서에 반영

## 반영 규칙

- 사용자 승인된 개선만 하네스 문서에 적용
- 하네스 문서 변경 시 관련 스킬 + CLAUDE.md 즉시 동기화
- 반영 후 변경 요약을 사용자에게 출력
