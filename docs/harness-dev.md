# Harness: 개발 단계

## 브랜치 전략

```
main
  └── feat/[project-name]          ← 통합 브랜치
       ├── feat/T-01-description   ← 태스크 브랜치
       ├── feat/T-02-description
       └── ...
  ← 최종 PR (통합 → main)
```

| 규칙 | 내용 |
|------|------|
| 통합 브랜치 | `feat/[project-name]`, main에서 생성 |
| 태스크 브랜치 | `feat/T-XX-description`, 통합 브랜치에서 생성 |
| 태스크 머지 | 태스크 → 통합 브랜치로 머지 |
| 최종 머지 | 통합 브랜치 → main (PR) |
| **main 직접 push** | **금지** — 반드시 PR 경유 |

## 레포 초기화 체크리스트

신규 레포 생성 시 아래 설정 적용. **기존 CI/보호 설정이 있는 레포는 스킵.**

| 항목 | 내용 | 템플릿 |
|------|------|--------|
| 브랜치 프로텍션 | main 직접 push 금지, PR 필수 | `gh api` 설정 |
| PR CI | 빌드 + 테스트 자동 실행, 실패 시 머지 차단 | [`templates/ci-pr.yml`](templates/ci-pr.yml) |
| 시크릿 스캔 | GitHub 기본 secret scanning + push protection | 레포 Settings |
| 머지 조건 | CI 전체 통과 필수 | 브랜치 프로텍션에 포함 |

**적용 판단:**
- `.github/workflows/` 에 CI 파일이 이미 있으면 → CI 템플릿 스킵
- 브랜치 프로텍션이 이미 설정되어 있으면 → 프로텍션 스킵

**머지 후 검증 (필수):**
- `git log [target-branch] --oneline -3`으로 커밋 반영 확인
- 머지 실패 시: 에이전트가 충돌 해결 시도, 불가 시 사용자 보고

## 태스크 분해

- Claude가 PRD 기능 요구사항 기반으로 자동 분해
- 태스크 목록 파일: `repositories/[project]/docs/tasks.md`

| 항목 | 내용 |
|------|------|
| ID | T-01, T-02... |
| 설명 | 구현 내용 |
| 상태 | Todo / In Progress / Done |
| 브랜치 | `feat/T-01-[description]` |

## 스펙 변경 시 플로우

기능 추가/변경/삭제 요청 시:
```
사용자 요청 + 질문으로 구체화
  ↓
PRD 문서 먼저 업데이트 (변경 부분)
  ↓
기획 리뷰 (변경 부분 대상)
  ↓
엔지니어링 리뷰 (변경 부분 대상)
  ↓
코드 작업 진행
```

**원칙:** 코드보다 문서가 먼저. 문서 없이 코드 수정 금지.
**리뷰 범위:** 전체 PRD가 아닌 변경된 부분에 한해서 기획/엔지니어링 리뷰 수행.

## 태스크별 개발 사이클

```
통합 브랜치에서 태스크 브랜치 생성
  ↓
개발 (Dev Agent, 1태스크만 위임)
  ↓
에이전트 결과 검증 (빌드 + 테스트 직접 실행)
  ↓ 실패 시 에이전트 재투입
유닛 테스트 작성 + 실행
  ↓ 실패 시 자동 개선 (최대 3회)
테스트 통과
  ↓
통합 브랜치로 머지 + 검증
```

**⛔ 절대 규칙 (위반 시 즉시 중단):**
- **에이전트당 1태스크만 위임** (여러 태스크 한번에 금지)
- 에이전트 완료 후 **빌드 + 테스트 직접 실행하여 검증** (에이전트 보고만 믿지 않음)
- 테스트 코드 없이 커밋/머지 금지
- 테스트 미통과 상태에서 다음 태스크 진행 금지
- 빌드 실패 상태에서 다음 태스크 진행 금지
- 이 규칙은 "급해서", "간단해서", "나중에" 등 어떤 이유로도 예외 불가

**테스트 규칙:**
- 비즈니스 로직·DB 접근 중심으로 작성 (trivial 코드 제외)
- 프론트엔드: 빌드 통과 + 컴포넌트 렌더링 테스트 필수
- 테스트 실패 시 Dev Agent가 자동 수정 (최대 3회)
- pre-commit/pre-push 훅: 테스트 미통과 시 커밋·푸시 차단

**각 태스크 완료 시 필수 체크:**
- [ ] 테스트 코드 작성됨
- [ ] 테스트 전체 통과
- [ ] 빌드 성공
- [ ] 통합 브랜치 머지 + git log 검증

**자동 진행 원칙:**
- 검증 통과 시 사용자에게 묻지 않고 다음 태스크 자동 진행
- 검증 실패 시에만 멈추고 보고
- 전체 태스크 완료 후 QA 단계로 자동 전환

## QA 사전 검증 (필수)

QA 진입 전 체크리스트:
- [ ] 전체 태스크 브랜치가 통합 브랜치에 머지 확인 (`git log`)
- [ ] 주요 소스 파일 존재 확인 (`ls src/`)
- [ ] 전체 테스트 통과 확인 (`pytest`)

## QA (QA Agent)

- 사전 검증 통과 후 실행
- PRD 기능 요구사항(F-XX) 기준 시나리오 테스트
- **웹 프로젝트: Playwright E2E 테스트 필수 실행**
  - 모든 페이지 접근 확인 (200/404)
  - **모든 네비게이션 링크 클릭 → 정상 페이지 도달 확인** (사이드바, 메뉴, 버튼 등)
  - 주요 사용자 플로우 자동화 테스트
  - headless 기본, `--headed`로 디버깅 가능
  - 프로젝트별 시나리오는 `e2e/` 디렉토리에 추가 확장 가능
- **UI 프로젝트: 디자인 QA (기능 QA와 함께 수행, 10점 만점, 평균 ≥ 8.0)**

  | 항목 | 설명 |
  |------|------|
  | 타이포그래피 | 텍스트 위계, 가독성, 줄간격 |
  | 컴포넌트 렌더링 | 테이블/인용문/코드블록 등 요소가 깔끔하게 표시 |
  | 여백·정렬 | 요소 간 간격, 정렬 일관성 |
  | 색상·대비 | 시맨틱 컬러 적용, 다크모드, 대비비 |
  | 반응형 | 모바일/태블릿/데스크톱 정상 표시 |
  | 인터랙션 | 호버/클릭 피드백, 전환 애니메이션 |
  | 접근성 | 터치 타겟 48px+, alt 텍스트, 키보드 접근 |

  - 판정: 평균 ≥ 8.0 → 통과, < 8.0 → 수정 후 재검증 (최대 3회)
  - → 디자인 원칙 상세: [`harness-design.md`](harness-design.md)

**QA 실패 시 처리:**
```
QA 실패 항목 발견
  ↓
Dev Agent에 실패 항목 재투입 → 수정
  ↓
테스트 통과 확인
  ↓
QA 재실행 (실패 항목만)
  ↓ 최대 3회 반복
3회 후에도 실패 → 사용자에게 보고
```

## 코드 리뷰 (Code Review Agent)

→ 평가 기준 상세: [`harness-code-review.md`](harness-code-review.md)

**평가 항목 (8개, 10점 만점):** 정확성 / 보안 / 성능 / 설계 / 가독성 / 테스트 / 동시성 / 운영 가시성

**PR 유형별 포커스:**

| PR 유형 | 집중 영역 |
|---------|-----------|
| 버그 픽스 | 정확성 + 테스트 |
| 새 기능 | 설계 + 보안 + 성능 |
| 리팩토링 | 가독성 + 테스트 |

**판정:**
- 평균 ≥ 8.0 → 통과
- 평균 < 8.0 → Dev Agent 재투입 (최대 3회)
- 3회 후 < 7.0 → 사용자에게 추가 사이클 여부 확인
- 3회 후 7.0~8.0 → 통과 처리

**발견 이슈 처리:**

| 심각도 | 처리 방식 |
|--------|-----------|
| High | Dev Agent가 자동 수정·반영 |
| Mid | Dev Agent가 자동 수정·반영 |
| Low | 전체를 테이블로 한번에 제시, 사용자가 번호로 선택 |
