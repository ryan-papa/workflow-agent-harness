## 배포 체크리스트 (배포 대상 프로젝트 필수)

배포가 수반되는 프로젝트는 README에 배포 체크리스트 섹션을 포함한다:

| 항목 | 예시 |
|------|------|
| 인프라 전제조건 | DB 스키마 생성, K8s namespace/secret |
| 환경변수·시크릿 | `.env` 필수 항목, K8s secret 목록 |
| 인증서·토큰 | SSL 인증서 경로, OAuth 토큰 파일 위치 |
| 외부 의존성 | 외부 API 키, 서드파티 서비스 설정 |
| 빌드·이미지 | Docker 빌드 명령, 이미지 import 방법 |
| 배포 명령 | Helm install/upgrade, kubectl apply |
| 배포 후 검증 | 헬스체크 URL, 로그 확인 명령 |

## 규칙

| 규칙 | 내용 |
|------|------|
| 작성 시점 | 프로젝트 초기화 시 템플릿 기반으로 생성 |
| 업데이트 | 기능 변경 시 README도 함께 업데이트 (PR 리뷰 시 체크) |
| 커맨드 검증 | README의 모든 커맨드는 실제 실행 가능해야 함 |
| 구조 일관성 | 같은 유형의 프로젝트는 같은 템플릿 구조 사용 |
| **push 전 검증** | **모든 push 전 README 검증 필수** → [`harness-ship.md`](harness-ship.md) 참조 |

## Push 전 README 검증

`rp-ship` 절차 중 push 직전에 자동 실행. 상세 기준은 [`harness-ship.md`](harness-ship.md) 참조.

| 항목 | 기준 |
|------|------|
| 검증 항목 | 프로젝트 개요, 디렉토리 구조, 설치/실행, 기능 목록, 기술 스택 (5항목) |
| 통과 기준 | 평균 8.0/10 이상 |
| 실패 시 | README 수정 → 재검증 (최대 3회) → 사용자 판단 위임 |
| **⛔ 절대 규칙** | **검증 미통과 시 push 차단** |

## 실전 팁

- README-Driven Development: 코드 전에 README 먼저 작성
- 복붙 테스트: 새 환경에서 README 커맨드 실행 검증
- 신규 팀원에게 읽혀보기: 질문 나오면 보강
- 섹션 7개 이상이면 목차 추가
