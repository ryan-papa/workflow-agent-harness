# Harness: PRD 단계

## 구체화 (Claude)

- 질문 1개씩 대화형 진행
- 기술 스택·구현 방식 불명확 시 자동 리서치
- **다단계 구체화:** 수집된 요구사항에서 구현 조건을 분해하고, 각 조건을 다시 검증
- **종료 조건:** 아래 필수 항목 수집 + 구현 조건 검증 완료 시 자동 종료

**구체화 프로세스:**
```
1차: 사용자 요구 수집 (무엇을 원하는가)
  ↓
2차: 구현 조건 분해 (이것을 하려면 무엇이 필요한가)
  ↓
3차: 각 조건별 검증 (가능한가, 제약은 없는가, 대안은)
  ↓
미충족 조건 발견 시 → 사용자에게 제약 + 대안 제시
```

**필수 항목 (2개 영역):**

도메인 정보 (대상·데이터·인증):

| 항목 | 예시 |
|------|------|
| 대상 특정 | 어떤 카페, 어떤 게시판, 어떤 사용자 등 |
| 데이터 구분 | 게시판별 역할 (공지/일반), 데이터 유형 (이미지/텍스트) |
| 인증·인가 | 로그인 방식, 토큰 발급 절차, 권한 범위 |
| 입력 데이터 | 기준 이미지 등록 방법, 초기 데이터 준비 |
| 결과 전달 대상 | 누구에게, 어떤 채널로, 어떤 포맷으로 |

기술 정보 (스택·환경):

| 항목 | 예시 |
|------|------|
| 핵심 기능 | 기능 목록 확정 |
| 개발 언어·런타임 | Python, Node.js 등 |
| 실행 환경 | 로컬, 클라우드 등 |
| 외부 연동 | API, 메시지 채널 등 |

UI/UX 정보 (웹/앱 프로젝트 시 필수):

| 항목 | 예시 |
|------|------|
| 대상 사용자 | 연령, 기기 (모바일/데스크톱) |
| 디자인 톤 | 밝은/어두운, 컬러, 분위기 |
| 레이아웃 | 주요 화면 구성, 네비게이션 |
| 인터랙션 | 애니메이션, 전환 효과, 피드백 |
| 참고 사이트 | 벤치마크, 좋아하는 디자인 예시 |

**질문 순서:** 도메인 정보 → 기술 정보 → **외부 API 인증 점검** (해당 시) → UI/UX (웹/앱 시) → 구현 조건 분해·검증

## 외부 API 인증 점검 (외부 API 연동 1개 이상 시 필수)

기술 정보 수집 완료 후, 외부 API 연동이 존재하면 PRD 작성 전에 아래 체크리스트를 순회한다.
**각 API별로** 아래 3개 영역을 질문으로 구체화한다.

### 1. 토큰 라이프사이클

| 항목 | 확인 내용 |
|------|----------|
| 발급 방식 | OAuth 플로우 / API 키 / 서비스 계정 등 |
| 토큰 유형 | access_token + refresh_token / API key only / JWT 등 |
| 만료 주기 | access_token 유효 시간, refresh_token 유효 시간 |
| 갱신 방식 | 자동 갱신 가능 여부, 갱신 시 새 refresh_token 발급 여부 |
| 갱신 시점 | 만료 전 사전 갱신 / 401 감지 후 갱신 / 둘 다 |
| 재시도 정책 | 갱신 실패 시 재시도 횟수, 영구 실패와 일시 실패 구분 |

### 2. 초기 셋업 + 장애 시나리오

| 항목 | 확인 내용 |
|------|----------|
| 최초 토큰 발급 | 수동 / 스크립트 자동화 / 관리 콘솔에서 발급 |
| redirect URI | 필요 시 사전 등록 여부, 로컬 개발용 URI |
| client secret | 필수 여부, 발급 위치 |
| 만료 알림 | refresh_token 만료 임박 시 알림 방식 (로그/메시지/이메일) |
| 알림 시점 | 만료 N일 전, 알림 빈도 (하루 1회 등) |
| 토큰 파일 손상 | 빈 파일/잘못된 JSON 시 복구 전략 |
| 동시 접근 | cron 중첩 실행 등 동시 쓰기 대응 (atomic write 등) |

### 3. 보안

| 항목 | 확인 내용 |
|------|----------|
| 시크릿 저장 위치 | `.env` / 별도 JSON / 환경변수 / 시크릿 매니저 |
| 토큰 저장 위치 | `.env`와 분리 여부, 파일 경로 |
| .gitignore | 시크릿·토큰 파일 모두 등록 여부 |
| 환경변수 분리 | 시크릿(변하지 않음) vs 토큰(주기적 갱신) 분리 관리 |
| 예시 파일 | `.env.example`, `config.example.yaml`에 실제 값 미포함 확인 |

## PRD 작성 (Doc Agent)

**섹션:**

| 섹션 | 내용 |
|------|------|
| 개요·목적 | 배경, 목표, 범위 |
| 사용자 스토리 | `As a ... I want ... So that ...` |
| 기능 요구사항 | 구체적 동작 정의 |
| 비기능 요구사항 | 성능, 보안, 확장성 |
| 제약사항 | 기술·비즈니스 제약 |
| Open Issues | TBD 항목 추적 |

**파일명:**
```
repositories/[project]/docs/prd/YYYYMMDD_HHMMSS_[feature]_[8자리랜덤].md
```
- `YYYYMMDD_HHMMSS`: `date +%Y%m%d_%H%M%S` 실행값
- 랜덤 문자열: 세션 시작 시 1회 생성 후 유지

## PRD 리뷰 (2단계 순차 실행)

```
Doc Agent 작성 완료
  ↓
[1단계] 기획 리뷰 (Planning Review Agent)
  ↓ 통과 (평균 ≥ 8.0)
[2단계] 엔지니어링 리뷰 (Engineering Review Agent)
  ↓ 통과 (평균 ≥ 8.0)
사용자 최종 확인
```

각 단계: 미달 시 Doc Agent 재작성, 최대 3회 반복

---

### 1단계: 기획 리뷰 (Planning Review Agent)

**평가 항목 (10점 만점):**

| 항목 | 설명 |
|------|------|
| 명확성 | 요구사항이 모호하지 않은가 |
| 완성도 | 누락된 섹션·내용 없는가 |
| 실현가능성 | 현실적으로 구현 가능한가 |
| 일관성 | 섹션 간 모순 없는가 |
| 측정가능성 | 성공 기준이 정량적인가 |
| 간결성 | 불필요한 내용 없이 핵심만 담겼는가 |
| 적정크기 | 문서 분량이 과도하지 않은가 |

---

### 2단계: 엔지니어링 리뷰 (Engineering Review Agent)

**평가 항목 (10점 만점):**

| 항목 | 설명 |
|------|------|
| 요구사항 명확성 | 비정량적 표현 없는가, 엣지 케이스 명시, Must/Should/Could 구분 |
| 기술적 실현가능성 | 현재 스택 정합성, 외부 의존성 확인, 데이터 모델 영향 |
| 범위·공수 적정성 | 범위 크리프 위험, MVP vs 전체 스펙 분리, 숨겨진 작업(인프라·모니터링·문서화) |
| NFR 충족도 | 성능 목표치, 보안(인증/인가/암호화), 확장성, 관측가능성(로깅/메트릭/알림) |
| 의존성·리스크 | 팀 간 의존성, 기술 부채 영향, 롤백 계획 |
| 테스트 가능성 | AC 존재 여부, 테스트 전략(단위/통합/E2E), QA 환경 유사성 |

**리뷰 원칙:** 각 항목에 "이걸 어떻게 테스트할 수 있는가?" 질문 적용

---

### 공통 규칙

**판정 (두 리뷰 모두 동일):**
- 평균 ≥ 8.0 → 통과
- 평균 < 8.0 → Doc Agent 재작성 (최대 3회)
- 3회 후 < 8.0 → `Max retry reached`, 사용자 판단

**결과 기록:** PRD 파일 하단 `## Review 결과`에 기획/엔지니어링 구분하여 기입

**규칙:** 작성·리뷰·개선은 자동 진행, 최종 문서만 사용자에게 전달

## PRD 상태

| 상태 | 조건 |
|------|------|
| `Draft` | 최초 작성 |
| `Planning Review` | 기획 리뷰 진행 중 |
| `Engineering Review` | 엔지니어링 리뷰 진행 중 |
| `Approved` | 두 리뷰 모두 통과 → 바로 개발 진입 |
| `Max retry reached` | 3회 후 미달 |
