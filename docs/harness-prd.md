# Harness: PRD 단계

## 구체화 (Claude)

- 질문 1개씩 대화형 진행
- 기술 스택·구현 방식 불명확 시 자동 리서치
- **다단계 구체화:** 수집된 요구사항에서 구현 조건을 분해하고, 각 조건을 다시 검증
- **종료 조건:** 아래 필수 항목 수집 + 구현 조건 검증 완료 시 자동 종료

**구체화 프로세스:**
```
1단계: 핵심 가치 (이 서비스가 주는 가치는 무엇인가)
  ↓
2단계: 사용자 흐름 (누가, 무엇을 입력하고, 무엇을 보고 싶은가)
  ↓
3단계: 화면 구성 (각 화면에 어떤 정보를, 어떤 형태로 보여줄 것인가)
  ↓
4단계: 시장 조사 (비슷한 서비스가 있는가, 차별점은 무엇인가)
  ↓
5단계: 재구체화 (시장 조사 결과 반영, 기능 보강/차별화)
  ↓
6단계: 기술 정보 + 구현 조건 검증
  ↓
미충족 조건 발견 시 → 사용자에게 제약 + 대안 제시
```

**필수 항목:**

1. 핵심 가치:

| 항목 | 질문 |
|------|------|
| 해결할 문제 | 사용자가 현재 겪는 불편함은? |
| 제공할 가치 | 이 서비스를 통해 사용자가 얻는 것은? |
| 핵심 경험 | 사용자가 "와, 좋다"라고 느끼는 순간은? |

2. 사용자 흐름 (입력 → 처리 → 출력):

| 항목 | 질문 |
|------|------|
| 입력 | 사용자가 무엇을 입력하는가 (각 입력 필드별 구체화) |
| 처리 | 입력된 데이터로 어떤 가공/분석을 하는가 |
| 출력 | 사용자에게 무엇을 보여주는가 (어떤 정보, 어떤 순서) |
| 후속 행동 | 결과를 본 후 사용자가 할 수 있는 행동은 (저장, 공유, 재검색 등) |

3. 화면 구성 (웹/앱 시 필수):

| 항목 | 질문 |
|------|------|
| 화면 목록 | 서비스에 필요한 화면은 몇 개인가 |
| 각 화면별 정보 | 어떤 정보를 어떤 형태(카드/리스트/지도/그래프)로 보여줄 것인가 |
| 정렬/필터 | 정보를 어떤 기준으로 정렬하고 필터링할 것인가 |
| 빈 상태/에러 | 결과가 없거나 오류 시 어떻게 보여줄 것인가 |
| 디자인 톤 | 분위기, 컬러, 참고 사이트 |
| 반응형 | 모바일/태블릿/데스크톱 대응 |

4. 시장 조사 (구체화 완료 후):
- 유사 서비스 검색 + 기능/화면/철학 분석
- 차별점 도출 → 재구체화에 반영

5. 기술 정보:

| 항목 | 예시 |
|------|------|
| 개발 언어·런타임 | Python, Node.js 등 |
| 실행 환경 | 로컬, 클라우드 등 |
| 외부 연동 | API, 메시지 채널 등 |
| 인증·인가 | 로그인 방식, 토큰 발급 |

**질문 순서:** 핵심 가치 → 사용자 흐름 → 화면 구성 → 시장 조사 → 재구체화 → 기술 정보

## PRD 작성 (Doc Agent)

**섹션:**

| 섹션 | 내용 |
|------|------|
| 개요·목적 | 배경, 목표, 범위 |
| 사용자 스토리 | `As a ... I want ... So that ...` |
| 기능 요구사항 | 구체적 동작 정의 |
| 비기능 요구사항 | 성능, 보안, 확장성 |
| 제약사항 | 기술·비즈니스 제약 |
| Open Issues | TBD 항목 추적 |

**파일명:**
```
repositories/[project]/docs/prd/YYYYMMDD_HHMMSS_[feature]_[8자리랜덤].md
```
- `YYYYMMDD_HHMMSS`: `date +%Y%m%d_%H%M%S` 실행값
- 랜덤 문자열: 세션 시작 시 1회 생성 후 유지

## PRD 리뷰 (2단계 순차 실행)

```
Doc Agent 작성 완료
  ↓
[1단계] 기획 리뷰 (Planning Review Agent)
  ↓ 통과 (평균 ≥ 8.0)
[2단계] 엔지니어링 리뷰 (Engineering Review Agent)
  ↓ 통과 (평균 ≥ 8.0)
사용자 최종 확인
```

각 단계: 미달 시 Doc Agent 재작성, 최대 3회 반복

---

### 1단계: 기획 리뷰 (Planning Review Agent)

**평가 항목 (10점 만점):**

| 항목 | 설명 |
|------|------|
| 명확성 | 요구사항이 모호하지 않은가 |
| 완성도 | 누락된 섹션·내용 없는가 |
| 실현가능성 | 현실적으로 구현 가능한가 |
| 일관성 | 섹션 간 모순 없는가 |
| 측정가능성 | 성공 기준이 정량적인가 |
| 간결성 | 불필요한 내용 없이 핵심만 담겼는가 |
| 적정크기 | 문서 분량이 과도하지 않은가 |

---

### 2단계: 엔지니어링 리뷰 (Engineering Review Agent)

**평가 항목 (10점 만점):**

| 항목 | 설명 |
|------|------|
| 요구사항 명확성 | 비정량적 표현 없는가, 엣지 케이스 명시, Must/Should/Could 구분 |
| 기술적 실현가능성 | 현재 스택 정합성, 외부 의존성 확인, 데이터 모델 영향 |
| 범위·공수 적정성 | 범위 크리프 위험, MVP vs 전체 스펙 분리, 숨겨진 작업(인프라·모니터링·문서화) |
| NFR 충족도 | 성능 목표치, 보안(인증/인가/암호화), 확장성, 관측가능성(로깅/메트릭/알림) |
| 의존성·리스크 | 팀 간 의존성, 기술 부채 영향, 롤백 계획 |
| 테스트 가능성 | AC 존재 여부, 테스트 전략(단위/통합/E2E), QA 환경 유사성 |

**리뷰 원칙:** 각 항목에 "이걸 어떻게 테스트할 수 있는가?" 질문 적용

---

### 공통 규칙

**판정 (두 리뷰 모두 동일):**
- 평균 ≥ 8.0 **+ 각 항목 ≥ 7** → 통과
- 평균 < 8.0 또는 항목별 최저 < 7 → Doc Agent 재작성 (최대 3회)
- 3회 후 미달 → `Max retry reached`, 사용자 판단

**결과 기록:** PRD 파일 하단 `## Review 결과`에 기획/엔지니어링 구분하여 기입

**규칙:** 작성·리뷰·개선은 자동 진행, 최종 문서만 사용자에게 전달

## PRD 상태

| 상태 | 조건 |
|------|------|
| `Draft` | 최초 작성 |
| `Planning Review` | 기획 리뷰 진행 중 |
| `Engineering Review` | 엔지니어링 리뷰 진행 중 |
| `Approved` | 두 리뷰 모두 통과 → 바로 개발 진입 |
| `Max retry reached` | 3회 후 미달 |
