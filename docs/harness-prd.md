# Harness: PRD 단계

## 구체화 (Claude)

- 질문 1개씩 대화형 진행
- 기술 스택·구현 방식 불명확 시 자동 리서치
- **종료 조건:** 5단계 질문 완료 + 후속 항목 수집 + 구현 조건 검증 완료 시 자동 종료
- **구체화 리트머스:** 서로 다른 두 사람이 이 문서만 보고 작업했을 때 같은 방향을 향하는가?

**구체화 5단계:**
```
1단계: 존재 이유 (왜 지금, 왜 이걸 해야 하는가)
  ↓
2단계: 사용자 (누가, 어떤 상황에서, 어떤 맥락으로)
  ↓
3단계: 핵심 장면 (가장 자주 일어날 장면 3개를 순서대로)
  ↓
4단계: 경계 (무엇이 아닌가, 포함/비포함)
  ↓
5단계: 성공 기준 (어떤 숫자/장면을 보면 성공인가, 접을 기준은)
  ↓
후속A: 화면·기능 구체화 (무엇을 만들까 — 화면·입출력·상태)
  ↓
후속B: 시장 조사 (유사 서비스 분석, 차별점 → 재구체화)
  ↓
후속C: 기술 정보 + 구현 조건 검증
  ↓
미충족 조건 발견 시 → 사용자에게 제약 + 대안 제시
```

**5단계 질문 항목:**

| 단계 | 묻는 것 | 대표 질문 |
|:----:|---------|----------|
| 1. 존재 이유 | 왜, 지금 왜 | "지금 이게 없어서 어떤 일이 안 풀리고 있나?" |
| 2. 사용자 | 누가 | "가장 자주 쓸 사람 한 명을 상황으로 묘사해주세요" |
| 3. 핵심 장면 | 무엇을, 어떻게 | "가장 자주 일어날 장면 3개를 순서대로요" |
| 4. 경계 | 무엇이 아닌가 | "'이런 느낌은 아니다' 싶은 레퍼런스가 있나?" |
| 5. 성공 기준 | 어떻게 알까 | "3개월 뒤 어떤 숫자/장면을 보면 성공? 반대로 접을 기준은?" |

**각 항목이 가져야 할 4가지 성질:**

| 성질 | 설명 |
|------|------|
| 관찰 가능 | 근거가 있다 |
| 반증 가능 | 확인할 방법이 있다 |
| 경계가 있다 | 포함/비포함이 분명하다 |
| 분기가 있다 | 예외·에러·빈 상태가 고려되어 있다 |

**후속 항목:**

1. 화면·기능 구체화:

| 항목 | 질문 |
|------|------|
| 화면 목록 | 필요한 화면은 몇 개인가 |
| 화면별 정보 | 어떤 정보를 어떤 형태(카드/리스트/지도/그래프)로 보여줄 것인가 |
| 입력·출력 | 사용자가 무엇을 입력하고, 무엇을 보는가 |
| 탐색 방식 | 사용자가 원하는 정보에 도달하는 경로는 (검색/필터/지도/추천 등) |
| 정렬·필터 | 정보를 어떤 기준으로 정렬·필터링하는가 |
| 데이터 소스 | 핵심 데이터를 어디서 확보하는가 (API/UGC/크롤링/직접 입력) |
| 추천·처리 로직 | 데이터를 어떤 기준으로 가공·추천하는가 (평점/개인화/트렌드 등) |
| 빈 상태·에러 | 결과 없음, 오류, 로딩 시 어떻게 보여주는가 |
| 디자인 톤 | 분위기, 컬러, 참고 사이트 |

2. 시장 조사:
   - 유사 서비스 검색 + 기능/화면/철학 분석
   - 차별점 도출 → 재구체화에 반영

3. 기술 정보:

| 항목 | 예시 |
|------|------|
| 개발 언어·런타임 | Python, Node.js 등 |
| 실행 환경 | 로컬, 클라우드 등 |
| 외부 연동 | API, 메시지 채널 등 |
| 인증·인가 | 로그인 방식, 토큰 발급 |

**질문 순서:** 존재 이유 → 사용자 → 핵심 장면 → 경계 → 성공 기준 → 화면·기능 구체화 → 시장 조사 → 기술 정보

## PRD 작성 (Doc Agent)

**섹션:**

| 섹션 | 내용 |
|------|------|
| 개요·목적 | 배경, 목표, 범위 |
| 사용자 스토리 | `As a ... I want ... So that ...` |
| 기능 요구사항 | 구체적 동작 정의 |
| 비기능 요구사항 | 성능, 보안, 확장성 |
| 제약사항 | 기술·비즈니스 제약 |
| Open Issues | TBD 항목 추적 |

**파일명:**
```
repositories/[project]/docs/prd/YYYYMMDD_HHMMSS_[feature]_[8자리랜덤].md
```
- `YYYYMMDD_HHMMSS`: `date +%Y%m%d_%H%M%S` 실행값
- 랜덤 문자열: 세션 시작 시 1회 생성 후 유지

### 간소 PRD (하네스 메타 변경용)

하네스 자체 문서(`docs/`, `CLAUDE.md`, 스킬, `.claude/settings.json`) 변경은 **간소 PRD** 허용 (완전 생략 금지).

| 필수 섹션 | 내용 |
|---------|------|
| 변경 이유 | 왜 이 메타 변경이 필요한가 |
| 영향 파일 | 변경되는 파일 목록과 역할 |
| 롤백 전략 | 문제 발생 시 원복 방법 |
| 검증 | CI·Claude 자체 점검·Codex 1회 리뷰 |

**파일명:** `docs/prd/[feature]/prd.md` (하네스 루트 기준)

## PRD 리뷰 (2단계 순차 실행)

```
Doc Agent 작성 완료
  ↓
[1단계] 기획 리뷰 (Planning Review Sub-Agent, Agent 툴)
  ↓ 통과 (평균 ≥ 8.0)
[2단계] 엔지니어링 리뷰 (Engineering Review Sub-Agent, Agent 툴)
  ↓ 통과 (평균 ≥ 8.0)
사용자 최종 확인
```

**⛔ 실행 주체**: 각 리뷰 단계는 **반드시 Agent 툴로 분리된 서브에이전트**가 수행한다. 메인 에이전트의 셀프 채점 금지. CLAUDE.md 하네스 절대 규칙.

각 단계 미달 시 Doc Agent 재작성, 최대 3회 반복 (매 회차 새 서브에이전트).

---

### 1단계: 기획 리뷰 (Planning Review Agent)

**평가 항목 (10점 만점):**

| 항목 | 설명 |
|------|------|
| 명확성 | 요구사항이 모호하지 않은가 |
| 완성도 | 누락된 섹션·내용 없는가 |
| 실현가능성 | 현실적으로 구현 가능한가 |
| 일관성 | 섹션 간 모순 없는가 |
| 측정가능성 | 성공 기준이 정량적인가, 실패 기준도 있는가 |
| 경계 명확성 | 스코프/비-스코프가 분명한가, 포함·비포함 판단이 가능한가 |
| 분기 충분성 | 예외·에러·빈 상태·대안 흐름이 고려되어 있는가 |
| 사용자 검증 게이트 | 실사용자 반응 검증 방법(프로토타입/인터뷰/A·B 테스트/내부 시범 사용) 명시 여부 |
| 대안 탐색 | 대안 A/B/C/D + 선택 근거 + 판단 근거 유형(창업자 직감/엔지니어 선호/제품 가설) 명시 여부 |

---

### 2단계: 엔지니어링 리뷰 (Engineering Review Agent)

**평가 항목 (10점 만점):**

| 항목 | 설명 |
|------|------|
| 요구사항 명확성 | 비정량적 표현 없는가, 엣지 케이스 명시, Must/Should/Could 구분 |
| 기술적 실현가능성 | 현재 스택 정합성, 외부 의존성 확인, 데이터 모델 영향 |
| 범위·공수 적정성 | 범위 크리프 위험, MVP vs 전체 스펙 분리, 숨겨진 작업(인프라·모니터링·문서화) |
| NFR 충족도 | 성능 목표치, 보안(인증/인가/암호화), 확장성, 관측가능성(로깅/메트릭/알림) |
| 의존성·리스크 | 팀 간 의존성, 기술 부채 영향, 롤백 계획 |
| 테스트 가능성 | AC 존재 여부, 테스트 전략(단위/통합/E2E), QA 환경 유사성 |

**리뷰 원칙:** 각 항목에 "이걸 어떻게 테스트할 수 있는가?" 질문 적용

---

### 공통 규칙

**판정 (두 리뷰 모두 동일):**
- 평균 ≥ 8.0 **+ 각 항목 ≥ 7** → 통과
- 평균 < 8.0 또는 항목별 최저 < 7 → Doc Agent 재작성 (최대 3회)
- 3회 후 미달 → `Max retry reached`, 사용자 판단

**결과 기록:** PRD 파일 하단 `## Review 결과`에 기획/엔지니어링 구분하여 기입

**규칙:** 작성·리뷰·개선은 자동 진행. 최종 문서만 사용자에게 전달.

## PRD 상태

| 상태 | 조건 |
|------|------|
| `Draft` | 최초 작성 |
| `Planning Review` | 기획 리뷰 진행 중 |
| `Engineering Review` | 엔지니어링 리뷰 진행 중 |
| `Approved` | 두 리뷰 모두 통과 → 바로 개발 진입 |
| `Max retry reached` | 3회 후 미달 |
