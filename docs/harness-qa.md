# Harness: QA + 콘텐츠 검수

## 프로젝트 유형별 분기

| 유형 | 판별 기준 | QA 방식 |
|------|----------|---------|
| 코드 프로젝트 | src/ 또는 실행 코드 존재 | 기능 QA + E2E + 디자인 QA |
| 콘텐츠 프로젝트 | MDX/MD 콘텐츠 중심 | 빌드 + 렌더링 + 내용 품질 검수 |

## QA 사전 검증 (필수)

- [ ] 전체 태스크 브랜치가 통합 브랜치에 머지 확인 (`git log`)
- [ ] 주요 소스 파일 존재 확인 (`ls src/`)
- [ ] 전체 테스트 통과 확인

## 코드 프로젝트 QA

- PRD 기능 요구사항(F-XX) 기준 시나리오 테스트
- **웹 프로젝트: Playwright E2E 테스트 필수 실행**
  - 모든 페이지 접근 확인 (200/404)
  - **모든 네비게이션 링크 클릭 → 정상 페이지 도달 확인**
  - 주요 사용자 플로우 자동화 테스트
  - headless 기본, `--headed`로 디버깅 가능
  - 프로젝트별 시나리오는 `e2e/` 디렉토리에 추가 확장 가능

### ⛔ 브라우저 수동 테스트 (필수, 코드 리딩만으로 대체 불가)

dev 서버를 실행하고 직접 curl 또는 브라우저로 확인:

| 항목 | 방법 |
|------|------|
| 모든 페이지 접근 | 각 페이지 URL에 curl 요청, 200 확인 |
| API 엔드포인트 | 임의의 데이터로 API 호출, 정상 응답 확인 |
| 버튼/링크 동작 | 모든 버튼 클릭 시 정상 동작 확인 |
| 입력 폼 | 임의의 데이터 입력 후 기능 동작 확인 |
| 외부 링크 | 외부 URL 응답 정상(200) 확인 |
| 에러 상태 | 잘못된 입력, 빈 입력 등 에러 처리 확인 |
| 환경변수 | API 키 등 설정이 정상 로딩되는지 확인 |

## 디자인 QA (UI 프로젝트)

기능 QA와 함께 수행. 각 10점 만점, 평균 >= 8.0 + 각 항목 >= 7 통과.

| 항목 | 설명 |
|------|------|
| 타이포그래피 | 텍스트 위계, 가독성, 줄간격 |
| 컴포넌트 렌더링 | 테이블/인용문/코드블록 등 요소가 깔끔하게 표시 |
| 여백·정렬 | 요소 간 간격, 정렬 일관성 |
| 색상·대비 | 시맨틱 컬러 적용, 다크모드, 대비비 |
| 반응형 | 모바일/태블릿/데스크톱 정상 표시 |
| 인터랙션 | 호버/클릭 피드백, 전환 애니메이션 |
| 접근성 | 터치 타겟 48px+, alt 텍스트, 키보드 접근 |

- 판정: 평균 >= 8.0 + 각 항목 >= 7 → 통과, 미달 → 수정 후 재검증 (최대 3회)
- → 디자인 원칙 상세: [`harness-design.md`](harness-design.md)

## 콘텐츠/문서 프로젝트 검수

코드가 아닌 콘텐츠(MDX/MD) 중심 프로젝트의 QA 대체 절차:

| 단계 | 내용 | 기준 |
|------|------|------|
| 빌드 검증 | `npm run build` 정상 완료 | 에러 0건 |
| 브라우저 렌더링 | dev 서버에서 변경 페이지 실제 확인 | 레이아웃 깨짐 없음 |
| 맞춤법 | 오탈자, 띄어쓰기 검수 | 발견 시 자동 수정 |
| 일관성 | 문체, 이모지 사용, 제목 형식 통일 | 기존 글과 동일 패턴 |
| 용어 통일 | 동일 개념에 다른 용어 사용 여부 | 용어사전 기준 일치 |

## QA/코드 리뷰 실패 시 재검증 플로우

```
QA 또는 코드 리뷰 미통과
  ↓
High/Mid/Low 이슈 수정 (코드 변경 발생)
  ↓
테스트 + 빌드 통과 확인
  ↓
코드 리뷰 재실행 (변경분 대상)
  ↓
QA 재실행 (변경분 영향받는 항목만)
  ↓ 최대 3회 반복
3회 후에도 미통과 → 사용자 보고
```

**원칙:**
- 코드가 변경되면 변경분에 대해 코드 리뷰 재실행 필수
- 코드 변경이 기능에 영향 주면 QA도 해당 항목 재실행
- 테스트 + 빌드 통과는 모든 단계 전환의 전제조건
